# Token Budget & Telemetry для субагентов

В проекте действует политика «скорость + качество > экономия токенов»
(см. `docs/agents-team-workflow.md`).
Это **не** означает «трать без счёта»: бесконтрольные циклы DA-fix-DA
или раздутые промпты всё равно нужно ловить.

Этот документ описывает бюджеты, телеметрию и правила эскалации.

## Сессионные бюджеты по классу задач

| Класс задачи                                  | Бюджет агентов   | Лимит на сессию              |
|------------------------------------------------|------------------|-------------------------------|
| Hotfix SEV1/SEV2 (B-NNN)                       | 1–3              | без лимита (приоритет — фикс) |
| Bug fix (B-NNN)                                | 2–4              | ≤ 200K input, ≤ 50K output    |
| Tech-debt cleanup (TD-NNN)                     | 2–4              | ≤ 300K input, ≤ 80K output    |
| Feature (F-NNN), effort ≤ 1 PD                 | 3–5              | ≤ 500K input, ≤ 150K output   |
| Feature (F-NNN), effort 1–3 PD                 | 5–8              | ≤ 1M input, ≤ 300K output     |
| RICE-оценка (3 голоса)                         | 3                | ≤ 100K input, ≤ 30K output    |

Если задача на старте предполагает превышение бюджета — это сигнал
**декомпозировать Issue**, не «закрыть глаза».

## Триггеры эскалации

Тимлид/оркестратор **обязан** запросить human-checkpoint при достижении одного из:

1. **Третий цикл DA** (см. `docs/agents-team-workflow.md` § 5).
   Bottom line: «застряли — нужно решение».
2. **Бюджет превышен на 50 %** относительно класса задачи.
   Тимлид/оркестратор останавливается, показывает что сделано, что осталось,
   просит решение: продолжить, декомпозировать, отложить.
3. **Один агент потребил > 50 %** общего бюджета задачи.
   Это симптом раздутого промпта или плохой декомпозиции.

## Телеметрия в TaskList / комментарии

После каждого `TeamDelete` тимлид публикует **cost report** в:
- комментарии Issue (если задача шла под Issue);
- `MEMORY.md` (если открытие нетипичного паттерна стоит закрепить).

Формат:

```markdown
## Cost report — F-042-export-pdf
- Команда: F-042-export-pdf
- Длительность: 1h 47m
- Teammates: 6
- Циклов DA: 2
- Tokens (sum):
  - Architect: 18.5K input / 4.2K output
  - SDE Backend: 124K input / 32K output
  - SDE Frontend: 89K input / 21K output
  - DBA: 31K input / 7K output
  - QA: 45K input / 12K output
  - Devil's Advocate: 67K input / 15K output  (×2 цикла)
- Total: 374K input / 91K output
- Estimated cost: $X.XX  (по тарифу модели)
- vs Budget (Feature 1–3 PD): 37 % / 30 % — в пределах
```

## Helper-скрипт

`./scripts/agents-cost-report.sh` — собирает данные после `TeamDelete`:

```bash
./scripts/agents-cost-report.sh F-042-export-pdf
```

Скрипт парсит TaskList (через TaskGet каждой задачи), агрегирует token usage,
формирует Markdown-отчёт. Тимлид вставляет в Issue или MR.

> Парсинг точных метрик зависит от версии Claude Code. Скрипт работает
> в режиме «best effort» и graceful-fallback: если поля недоступны,
> выводит структуру отчёта с предупреждением и оставляет ручное заполнение.

## Анти-паттерны

- ❌ «Я просто потрачу больше токенов, и всё получится». Симптом проблемы
  с декомпозицией. Возьмите паузу, разбейте Issue.
- ❌ Один большой промпт со всем контекстом всем агентам. Каждому
  агенту — таргетированный контекст под его задачу.
- ❌ Игнорировать триггер эскалации, потому что «у пользователя нет под рукой».
  Альтернатива — полная остановка с резюме до возвращения пользователя.
- ❌ Пропустить cost report после крупной задачи. Без него нет данных
  для калибровки бюджетов.

## Калибровка бюджетов

Раз в 2 недели — review последних 10 cost reports:
- Систематические перерасходы → пересмотр бюджета или декомпозиции Issues.
- Систематические недорасходы → ужесточение бюджета или пересмотр класса.

Issue: `TD-` с label `process`, milestone «Process review».
