# dps150-web — дизайн системы

Статус: согласован с владельцем проекта (гриль-сессия 2026-07-14).
Все решения ниже зафиксированы по итогам интервью; изменения — через ADR.

## 1. Что это

Веб-приложение для полного управления лабораторным блоком питания
**FNIRSI DPS-150** (USB-CDC serial), развёрнутое в домашнем Talos k8s кластере:

1. Полное управление БП: уставки V/I, включение выхода, уставки защит
   (OVP/OCP/OPP/OTP/LVP), яркость/звук, аппаратные пресеты M1–M6.
2. Профили (имя + V + I + уставки защит) с хранением в БД и назначением
   на аппаратные ячейки M1–M6.
3. Визуализация: живые показания (V/I/P, входное напряжение, температура,
   CC/CV, состояние защит) + исторические графики за месяц.

Проект пишется с прицелом на open source (MIT, README на английском,
никаких домашних адресов/секретов в коде — только в values/CI-переменных).

## 2. Устройство и протокол

- FNIRSI DPS-150: 0–30 В / 0–5 А / 150 Вт, USB-C (CDC serial, 115200 8N1).
- Бинарный протокол «memory-mapped registers», little-endian, float32 IEEE-754.
  Полный reverse-engineered референс: https://github.com/cho45/fnirsi-dps-150
  (docs/FNIRSI_DPS-150_Protocol.md), исходная CLI-реализация (подмножество):
  https://github.com/svenk123/dps150tool.
- Кадр: `F1|F0 <GROUP> <REG> <LEN> <DATA…> <CHK>`, CHK = (REG+LEN+ΣDATA)&0xFF.
- Ключевые регистры: C1/C2 уставки V/I; DB выход (0/1); C5–D0 пресеты M1–M6;
  D1–D5 уставки защит; D6/D7 яркость/звук; C0/C3/C4/E2/E3 телеметрия
  (устройство пушит сама каждые 500 мс); D8/D9/DA учёт Ач/Втч;
  DC срабатывание защиты (OK/OVP/OCP/OPP/OTP/LVP/REP); DD режим CC/CV;
  FF полный дамп состояния (139 байт).
- Сессия: enable (`F1 C1 00 01 01 02`) → baud → работа → disable.

### 3.1 Транспорт устройства (ключевая абстракция)

Драйвер DPS-150 в бэкенде работает поверх интерфейса транспорта
(`io.ReadWriteCloser` + reconnect-семантика). Три реализации,
выбор — конфигом (`DPS_TRANSPORT`):

| Транспорт | URI | Использование |
|---|---|---|
| serial | `serial:///dev/ttyUSB0` (или by-id) | локальный запуск на linux/macos, БП воткнут напрямую |
| tcp | `tcp://10.20.0.5:2150` | прод в кластере, через ser2net на pve |
| mock | `mock://` | эмулятор устройства: e2e в CI, разработка без железа |

Требование владельца (зафиксировано отдельно): серверная часть обязана
одинаково работать и с tcp-сокетом через ser2net, и с последовательным
портом напрямую.

Эмулятор реализует протокол на уровне кадров: телеметрия 2 Гц, реакция на
записи регистров, имитация срабатывания защит — тот же бинарь,
включается конфигом.

### 3.2 Бэкенд (Go)

- Единственный владелец соединения с устройством (порт одноклиентский);
  все потребители (REST, WS, история, правила) работают через внутренний hub.
- REST API: уставки, выход, защиты, профили, история, устройство/статус.
- WebSocket: живая телеметрия 2 Гц + события (защиты, CC/CV, connect/disconnect).
- Писатель истории: батчами в БД; ретенция и минутные агрегаты — фоновые джобы.
- Telegram-уведомления: сработка защиты, потеря/восстановление связи
  с устройством, (Этап 3) события автостопа. Настройка типов — в UI.
- Метрики Prometheus `/metrics` (+ ServiceMonitor) — подхватываются
  kube-prometheus-stack, смотреть в SigNoz.
- При потере связи: экспоненциальный reconnect, статус в UI/WS, событие в журнал.

### 3.3 Фронтенд (React + TS + Vite + Ant Design + TanStack Query)

- Главный экран: крупные показания V/I/P (+ CC/CV, вход, температура),
  переключатель выхода (с подтверждением), быстрые уставки, применение профиля.
- Графики: uPlot (живое окно + история из БД с зумом/панорамой).
- Страницы: Dashboard, History, Profiles, Events (журнал), Settings.
- Адаптив: десктоп — полный UI; телефон — бенч-режим главного экрана
  (крупные цифры, выключатель, профиль) с первого этапа.
- i18n: react-i18next, локали ru+en с первого экрана.
- Realtime — WebSocket, данные/мутации — TanStack Query поверх REST.

### 3.4 Хранение

- **Прод**: PostgreSQL 17 в общем CNPG `pg-cluster` по паттерну partdb:
  managed role `dps150` + `Database` CR + `VaultStaticSecret`
  (Vault: `secret/pg-cluster/dps150/database`); подключение через
  `pg-cluster-pooler-rw.pg-cluster.svc:5432`, миграции — через
  `pg-cluster-direct` (мимо пулера).
- **Локально**: SQLite (pure-Go, без cgo: GORM-драйвер `glebarez/sqlite`
  поверх `modernc.org/sqlite`) — один бинарь без зависимостей.
- Конфигурация хранилища (контракт): `DPS_DB_DRIVER` = `sqlite` (default) |
  `postgres`; `DPS_DB_DSN` = путь к файлу (default `dps150.db`) | postgres-DSN.
  Недоступная БД не валит приложение: фоновый reconnect, storage-фичи
  отвечают 503, управление устройством работает.
- Схема переносимая: время — unix millis (integer), без диалектных функций;
  агрегация по времени — целочисленным делением timestamp.
- Данные: `samples` (2 Гц, ретенция 30 дней), `samples_1m` (минутные агрегаты
  min/avg/max, ретенция 1 год), `profiles`, `events` (защиты, connect/disconnect,
  включения выхода — кто/когда/что), `settings`, (Этап 3) `automation_rules`.
- Объём: ~5,2 млн строк/мес — партиции/индексы по времени, без TSDB.

### 3.5 Безопасность

- Ingress `dps150.r2bnj.ru` целиком за Authelia forward-auth
  (middleware `authelia-forwardauth-authelia@kubernetescrd`), SSO кластера.
- ser2net на pve: слушает только для подсети кластера (файрвол/bind).
- Секреты (PG, Telegram token) — Vault → VaultStaticSecret → k8s Secret.
- Применение профиля НИКОГДА не включает выход автоматически.
  Включение выхода — только явное действие с подтверждением в UI.
- (Этап 3) API-токены для скриптового доступа.

## 4. Деплой и окружения

- **Одно окружение** (ns `dps150`): второй инстанс не смог бы подключиться
  к одноклиентскому ser2net; e2e и разработка — на эмуляторе.
- GitLab CI (`gitlab.zztopper.ru`, registry `:5005`): lint → test → build;
  CI только собирает и пушит образы (:short-sha + :latest на master).
- Деплой — GitOps через ArgoCD (ADR-005): чарт живёт в
  `infrastructure/argocd-platform` `apps/dps150/`, релиз = MR с бампом
  `image.tag`; ApplicationSet создаёт namespace с PSA-лейблами, selfHeal.
- Кластерные факты: dnsDomain `k8s.r2bnj.ru` (не cluster.local!), ingress-class
  `traefik`, cert-manager ClusterIssuer `letsencrypt-cloudflare`,
  external-dns → Cloudflare, storage `longhorn` (для БД не нужен — БД в CNPG).
- Локальный запуск: бинарь (serial+SQLite), docker-compose
  (backend+frontend+postgres, для docker/portainer/orbstack).

## 5. Пользователь и сценарии (CJM)

Единственный пользователь — владелец. Устройства: десктоп за верстаком
+ телефон из другой комнаты.

| Сценарий | Путь | Что критично |
|---|---|---|
| Быстро запитать схему | открыл dashboard → профиль или уставки → выход ON → смотрит потребление | скорость, крупные цифры, 1 клик до профиля |
| Длительный прогон | выставил → ушёл → следит с телефона → Telegram при сработке защиты | графики, события, уведомления, мобильный экран |
| Заряд/тест АКБ | профиль заряда → следит за Ач/Втч → (Этап 3) автостоп по условию | счётчики D8/D9/DA, автостоп-правила, Telegram |
| Ремонт/диагностика | профиль с жёсткими лимитами тока → подал питание → мгновенно видит CC/сработку | защиты в профиле, быстрая реакция UI на события |

## 6. Этапы (= GitLab Milestones)

- **Этап 1 «PoC»**: скаффолд (Go+React+CI+Docker+Helm), протокол-библиотека
  + эмулятор + транспорты serial/tcp/mock, минимальный веб (живая телеметрия,
  V/I, выход), SQLite локально, ser2net на pve, БД в CNPG, Authelia,
  автодеплой в кластер. Результат: сервис живёт в кластере и управляет БП.
- **Этап 2 «MVP»**: профили + синк M1–M6, история за месяц + графики,
  уставки защит + журнал событий, Telegram-уведомления, мобильная вёрстка,
  счётчики Ач/Втч.
- **Этап 3 «v1.0»**: правила автостопа (ток < X дольше Y мин / после Z Ач/Втч /
  через T часов; подстраховка — аппаратные защиты), экспорт CSV, API-токены,
  публикация зеркала на GitHub.

## 7. Открытые точки

- Точное имя serial-устройства на pve (`/dev/serial/by-id/…`) — узнаем при
  подключении БП; ser2net настраивается по SSH (root@10.20.0.5) с
  подтверждением перед изменениями.
- Managed role + Database CR + VSS для dps150 добавляются в репо
  `k8s-talos-cluster` (helm/pgsql-cluster) — отдельный MR туда.
- CI-переменная `KCONFIG` — проверить наличие на уровне группы applications,
  иначе скопировать из проекта cattery.
