# Agents Team Workflow (TeamCreate + tmux)

Workflow многоагентной команды через `TeamCreate` + tmux-окно с разделением
на пейны. Главный Claude в pane-0 — **тимлид**; teammates spawn-ятся
в свободные пейны и работают параллельно.

## Зачем

Параллелизация: пока один агент пишет backend, другой — frontend, третий —
тесты, четвёртый — миграции. Тимлид координирует, агенты обмениваются
сообщениями через `SendMessage`, а Devil's Advocate выполняет финальную
верификацию **до** коммита.

## Жизненный цикл

```
TeamCreate                       # создать команду (получаешь team_name)
  TaskCreate × N                 # все задачи сразу с зависимостями
    Task × M (одновременно)      # spawn teammates в одном сообщении
    SendMessage(...)             # обмен между teammates
  TaskList                       # отслеживать прогресс
  Task(devils-advocate, ...)     # ОБЯЗАТЕЛЬНО — DA-верификация
  Iterate fixes                  # устранение находок DA до коммита
TeamDelete
```

## Принципы тимлида (главный Claude в pane-0)

### 1. Максимальный параллелизм по умолчанию

Скорость и качество **приоритетнее экономии токенов**. Пользователь сознательно
оплачивает параллелизм — задача тимлида использовать его полностью.

**Правила:**
- Декомпозируй задачу на независимые подзадачи **до** начала реализации.
- Запускай teammates **одним сообщением** с несколькими `Task` tool calls —
  только так Claude Code запускает их параллельно.
- Предпочитай **ширину** (3 параллельных агента) **глубине** (3 последовательных).
- Если зависимостей нет — стартовать всех сразу. Если есть —
  использовать `addBlockedBy` в `TaskCreate`, а не последовательность ожиданий.
- 5–8 одновременных teammates — норма для крупной фичи; меньше — это
  чаще всего недодекомпозированная задача.

**Анти-паттерны (не делай так):**
- ❌ Запустить backend-агента, дождаться, запустить frontend-агента.
- ❌ «Сэкономлю токены — пусть один агент сделает всё» —
  результат хуже и медленнее.
- ❌ Самому реализовывать в pane-0 то, что можно делегировать.

### 2. Состав команды — под задачу

Тимлид сам выбирает набор ролей под конкретную задачу. Минимум:
- Реализаторы (Backend / Frontend / DBA / DevOps — те, что нужны).
- **Devil's Advocate** — обязателен в любом раскладе как финальный gate.

Дополнительно по необходимости:
- Architect — если затрагивается архитектура / нужен ADR.
- System Analyst — если требования не до конца формализованы.
- QA — если нужны новые тесты или E2E-сценарии.
- Technical Writer — если меняются API / docs / CHANGELOG.

### 3. Модели

Выбор по **характеру задачи**, не по роли:

| Класс задачи                                                              | Модель    |
|---------------------------------------------------------------------------|-----------|
| **Default**: реализация, рефакторинг, ревью, архитектура, спецификации,   | **opus**  |
| отладка, оценки, тесты с логикой. Все ключевые роли (Architect, System    |           |
| Analyst, Devil's Advocate, SDE/DBA/QA/DevOps/SRE, Technical Writer и т.д.)|           |
| Исследование без принятия решений: чтение/поиск кода, сбор фактов,        | **haiku** |
| инвентаризация, навигация по большому объёму материала, Explore-сценарии  |           |
| Тривиальные исполнительские: переименование, форматирование, генерация    | **sonnet**|
| повторяющихся файлов по жёсткому шаблону, прямой перевод                  |           |

Если есть **выбор или оценка** — задача не тривиальная, берётся **opus**.
Указывай `model` в каждом `Task`-вызове явно.

### 4. ОБЯЗАТЕЛЬНАЯ верификация Devil's Advocate

После того как реализаторы доложили результат, **до коммита** тимлид
запускает Devil's Advocate с `model: opus`:

```
Task(
  subagent_type="devils-advocate",
  team_name="<team>",
  model="opus",
  prompt="""
    Проверь реализацию <задача>.
    Изменения в коммитах: <git diff / список файлов>.
    Issue: <PREFIX-NNN>.
    AC: <ссылка>.

    Найди:
    - CRITICAL — баги/уязвимости/contract mismatches, без фикса в прод нельзя.
    - HIGH — серьёзные риски, фикс обязателен до мерджа.
    - MEDIUM — должно быть учтено, обоснуй отказ если не делаешь.
    - LOW — минорные замечания.
    - WISHES — улучшения и пожелания.
  """
)
```

**Гейтинг-правило:** коммит/мердж разрешены только когда:
- ✅ Все CRITICAL устранены **в текущей сессии**.
- ✅ Все HIGH устранены **в текущей сессии**.
- ✅ **Каждая** оставшаяся находка (MEDIUM/LOW/WISHES, pre-existing,
  out-of-scope, любой иной случай неустранения в сессии) — оформлена как
  **новый Issue** в трекере (`./scripts/issue-create.sh`).
  Без исключений.

> «Находка без следа» запрещена. Не «устно решили отложить», не «может потом» —
> только Issue с обоснованием в трекере. Аудит-след обязан сохраниться.

В DA-отчёте секция «Tracked» содержит таблицу: каждая не-устранённая находка
↔ ссылка на Issue. Тимлид перед коммитом проверяет: каждая строка отчёта
либо устранена в коде, либо имеет Issue-ссылку.

### 5. Цикл «найди → исправь → перепроверь» с лимитом

Если DA нашёл проблемы:
1. Тимлид декомпозирует находки на задачи.
2. Запускает реализаторов параллельно (повторно SDE / DBA / DevOps).
3. **Снова запускает DA** для верификации фиксов.
4. Цикл повторяется до зелёного отчёта DA.

Это часть нормального workflow, не «провал». Сильная задача проходит 1–3 цикла.

**Жёсткий лимит: 3 цикла.** Если после третьего DA-прогона остаются
неустранённые CRITICAL/HIGH:
- Тимлид **обязан** остановиться и эскалировать пользователю.
- Резюме передаваемое пользователю должно содержать:
  - что устранено;
  - что осталось и почему;
  - оценку стоимости продолжения (token + время);
  - предложение декомпозиции на меньшие Issues (часто корень — слишком
    большая Issue).
- Без human-checkpoint двигаться дальше **запрещено**.

### 6. Human checkpoints — обязательные точки согласия с пользователем

Тимлид паузится и **ждёт** явного «продолжай» от пользователя в этих точках:

1. **После Architect/System Analyst — до начала реализации.**
   Тимлид показывает: план задач, ADR (proposed), LikeC4-обновления,
   декомпозицию TaskCreate. Пользователь подтверждает направление или
   корректирует. Это спасает от полной реализации в неверном направлении.
2. **После DA-Verdict APPROVED — до push в remote.**
   Тимлид показывает: что сделано, какие Issues созданы для отложенных
   находок, ссылки на ADR/LikeC4. Пользователь даёт `git push`.
3. **На 3-м неуспешном DA-цикле** (см. п. 5).

Между check'ами агенты работают автономно с максимальным параллелизмом.
Check'и — короткие, не превращайте их в обсуждения. Цель: пользователь
видит траекторию и может развернуть до того, как потрачено много токенов.

### 6. ADR-чек

Если в ходе работы принято архитектурное решение или сделан non-trivial выбор —
тимлид обязан запустить Architect или System Analyst для публикации ADR
**до** закрытия Issue. См. `docs/adr-workflow.md`.

## Шаблон сессии: реализация фичи F-042

```
1. TeamCreate(name="F-042-export-pdf")

2. TaskCreate × N сразу со всеми задачами и зависимостями:
   - "Спроектировать API экспорта"           (Architect)         → blocks all
   - "Реализовать backend handler"           (SDE Backend)       → blockedBy Arch
   - "Реализовать UI экспорта"               (SDE Frontend)      → blockedBy Arch
   - "Миграция для аудита экспорта"          (DBA)               → blockedBy Arch
   - "Юнит-тесты + E2E"                      (QA)                → blockedBy backend, frontend
   - "Обновить CHANGELOG, README, Swagger"   (Technical Writer)  → blockedBy backend
   - "ADR-NNN: Выбор PDF-движка"             (Architect)         → blockedBy Arch
   - "Devil's Advocate review"               (Devil's Advocate)  → blockedBy ALL

3. Task × 1 (Architect) — сначала архитектура и ADR (model: opus).

4. Получив результат — одним сообщением Task × 4 параллельно:
   Backend / Frontend / DBA / Tech Writer (model: opus).

5. Task QA (model: opus) после готовности backend+frontend.

6. Task Devil's Advocate (model: opus) — последним, видит всю работу.

7. Если DA нашёл проблемы:
   - Распределить fixes по реализаторам (параллельно).
   - Снова Task Devil's Advocate.
8. Когда DA даёт зелёный отчёт — коммит, MR, закрытие Issue.

9. TeamDelete.
```

## Шаблон сессии: RICE-оценка

Полностью параллельная — все три голоса независимы.

```
1. TeamCreate(name="rice-F-042")

2. TaskCreate × 3 (без взаимных зависимостей).

3. Одно сообщение, три Task'а одновременно (все model: opus):
   - subagent_type="architect"
   - subagent_type="system-analyst"
   - subagent_type="devils-advocate"
   prompt — одинаковая постановка с полным контекстом Issue.

4. TaskList → ждём завершения всех трёх.

5. Тимлид сводит таблицу, обновляет описание Issue в трекере.

6. TeamDelete.
```

## Tmux

Запуск:
```bash
./scripts/tmux-team.sh F-042 6      # 6 пейнов = тимлид + 5 teammates
```

Hotkeys (`Ctrl-b` затем):
- стрелки / `o` — переключение пейнов;
- `z` — fullscreen toggle активного pane;
- `d` — detach (сессия живёт);
- `[` — режим скроллинга (выход — `q`).

`tmux attach -t F-042` — вернуться. `./scripts/tmux-kill.sh F-042` — закрыть.

## Когда команда не нужна

- Простая правка одного файла (один SDE-агент или сам pane-0).
- Pure-research задача (Explore-агент через Task tool).
- Тривиальный bugfix без архитектурных последствий — допустим pane-0
  + единичный Devil's Advocate-pass перед коммитом.

## Troubleshooting

- **Teammate не появляется** — проверь, что `subagent_type` совпадает
  с именем файла в `.claude/skills/` (без `.md`).
- **Все запустились, но pane'ов мало** — Claude Code открывает teammates
  в новых окнах, используй `Ctrl-b w` для списка.
- **DA постоянно находит то же самое** — реализатор игнорирует находки;
  проверь, что цикл «найди → исправь → перепроверь» действительно идёт.
- **Долго работает** — это нормально для сложной фичи; параллелизм даёт
  выигрыш по wall-clock, но каждая Task всё ещё думает.
