# Definition of Ready (DoR)

Issue **готова к старту** только когда все пункты ниже отмечены.
Если хоть один не выполнен — Issue остаётся в `status::backlog` или
`status::needs-grooming`. Старт работы по неготовой Issue — анти-паттерн,
ведущий к scope creep, переделкам и красным DA-отчётам.

## Чек-лист DoR

### A. Бизнес-контекст
- [ ] Заполнен раздел **Бизнес-контекст** (зачем, какую боль решает, для кого).
- [ ] Указано **Out of scope** явно (что НЕ делаем в рамках Issue).
- [ ] Stakeholder идентифицирован (для F-* — продакт/заказчик; для B-* —
      тот, кто увидит фикс; для TD-* — техлид/архитектор).

### B. Acceptance Criteria
- [ ] Минимум 1 AC сформулирован в формате Given/When/Then или явного чек-листа.
- [ ] Каждый AC проверяем (есть способ binary verify: «выполнено / не выполнено»).
- [ ] AC не содержат «или», «может быть», «желательно» — только конкретику.

### C. Technical readiness
- [ ] Зависимости названы и **разрешены** (не ждут другую in-progress Issue,
      кроме случаев когда blockedBy явно указан).
- [ ] Если Issue требует архитектурного решения — соответствующий ADR
      открыт как Proposed (или зафиксировано, что будет создан в первый день).
- [ ] Если Issue затрагивает БД — намечена стратегия миграции (см. `docs/migration-safety.md`).
- [ ] Если Issue user-видимая — требование E2E-сценария включено в AC.

### D. Estimation & priority
- [ ] **RICE-оценка** проведена тремя голосами (Architect, System Analyst,
      Devil's Advocate) — см. `docs/rice-scoring.md`.
- [ ] Effort ≤ 3 person-days (или Issue декомпозирована).
- [ ] Issue привязана к Milestone.
- [ ] Labels проставлены: `feature`/`bug`/`tech-debt`/..., подсистема
      (`backend`/`frontend`/...).

### E. Process readiness
- [ ] Заголовок в формате `PREFIX-NNN: <title>` (полученный через
      `./scripts/issue-counter.sh`).
- [ ] Описание соответствует template'у (`templates/.gitlab/issue-templates/`
      или `templates/.github/ISSUE_TEMPLATE/`).
- [ ] Назначен исполнитель (агент/команда).

## Анти-паттерны

- ❌ «Подсчитаем эффорт после первого дня работы.» — нарушение, ломает планирование.
- ❌ «AC уточним в процессе.» — гарантирует переделку.
- ❌ «Архитектурное решение обсудим во время реализации.» — обсуждение
      перерастает в рефакторинг и блокирует MR.
- ❌ «Зависимости разрешим параллельно.» — формально допустимо через
      `addBlockedBy`, но если зависимость — ждущий чужой ответ → не Ready.

## Кто проверяет DoR

- При создании Issue — System Analyst (если есть) или оркестратор.
- При взятии в работу — оркестратор: открывает Issue, проверяет чек-лист.
  Если что-то отсутствует — добавляет недостающее (или
  отправляет обратно в grooming).

## Связь с Definition of Done

DoR — про вход. DoD (`docs/definition-of-done.md`) — про выход. Если DoR
плохой, DoD недостижим: невозможно проверить выполнение AC, которые сами
размыты.
