# Migration Safety

Принципы безопасной миграции БД. Применяются к любому изменению схемы:
DDL (ALTER, ADD COLUMN, DROP), DML на больших таблицах, изменение индексов.

## Базовое правило: expand → migrate data → contract

Никаких «atomic deploy миграция + код». Любое breaking-change-изменение схемы
делается **в три фазы**:

```
1. EXPAND
   - Добавить новые поля/таблицы/индексы; старые сохранить.
   - Деплой кода, который умеет работать со ОБЕМИ версиями схемы.

2. MIGRATE DATA (опционально)
   - Backfill, копирование данных из старого формата в новый.
   - Идемпотентно, в батчах, без локов.
   - Может занять часы/дни на больших таблицах.

3. CONTRACT
   - Удалить старые поля/таблицы.
   - Деплой кода, который использует только новый формат.
```

Между фазами — boundary, переход к следующей фазе только когда первая
полностью завершена и проверена в проде.

## Запреты

- ❌ `ALTER TABLE ... ADD COLUMN ... NOT NULL DEFAULT '<значение>'` на большой
  таблице без прицела. Может заблокировать таблицу. Решение:
  ADD COLUMN nullable → backfill → SET NOT NULL.
- ❌ DROP COLUMN или RENAME COLUMN сразу. Решение: 2 деплоя (expand-contract).
- ❌ `CREATE INDEX` без `CONCURRENTLY` (Postgres) — лочит таблицу.
- ❌ Длинные транзакции на горячих таблицах.
- ❌ Откат миграции без проверки совместимости с прод-кодом
  (если код уже использует новые поля — откат сломает прод).

## Чек-лист DBA-агента перед merge

- [ ] Миграция тестируется и в локали, и на staging-окружении на копии prod-данных.
- [ ] Предусмотрен `down`-миграционный скрипт И **проверен в реальности**
      (просто наличие файла недостаточно).
- [ ] Long-running операции вынесены в backfill-job, не в `up.sql`.
- [ ] Создание индексов — `CONCURRENTLY` (Postgres) / equivalents.
- [ ] Lock-time оценен: на проде ALTER не должен превышать 1 секунду
      (а лучше 100 мс).
- [ ] Совместимость с **обеими** версиями кода (текущей и новой) подтверждена.
- [ ] Для больших миграций — план батчирования (например, 10k строк × N батчей).
- [ ] CI-pipeline для миграций имеет dry-run на staging перед prod.

## Документация миграции

В каждом `up.sql` / `down.sql` — комментарий:

```sql
-- Migration NNNN: <название>
-- Phase: expand | migrate-data | contract
-- Related Issue: F-NNN
-- ADR (если применимо): ADR-NNN
-- Lock estimate: <length>
-- Backfill: <runtime estimate>
-- Compatibility: совместимо с кодом версии vX.Y и vX.Y+1
-- Rollback: <safe condition>
```

## Большие данные

Для backfill > 1M строк:
- Job через очередь / cron, а не SQL-скрипт.
- Прогресс-репортинг (offset, last-id).
- Ограничение пропускной способности, чтобы не убить prod-latency.
- Метрика: «% мигрированных строк», алерт при стопе/ошибке.

## Online schema migrations

Для MySQL: `gh-ost`, `pt-online-schema-change`.
Для Postgres: `pg_repack` для перестройки больших таблиц.
Использовать, если Vacuum/Reindex в любом виде вызывают unacceptable lock.

## Связь с release process

Каждая миграция — отдельный коммит / отдельный MR (не сваливать с feature-кодом).
Если фича требует expand-contract — две Issue (expand + contract), одна за другой.
Между ними — отдельный релиз с проверкой стабильности на проде.
