# RICE Scoring

Каждый новый Issue получает RICE-оценку **тремя независимыми агентами**:
**Architect**, **System Analyst**, **Devil's Advocate**. Финальный RICE —
среднее трёх оценок.

## Параметры

| Параметр       | Шкала                          | Описание                                                        |
|----------------|--------------------------------|-----------------------------------------------------------------|
| **Reach**      | 1–5                            | Сколько пользователей/кейсов затронет за квартал                |
| **Impact**     | 0.25 / 0.5 / 1 / 2 / 3         | Степень влияния на ключевые метрики                             |
| **Confidence**| 0–1 (0.5 / 0.7 / 0.9 / 1.0)     | Насколько уверены в оценке (есть ли данные)                     |
| **Effort**     | person-months                  | Затраты в человеко-месяцах (1 PM ≈ 4 рабочие недели)            |

Формула: `RICE = (R × I × C) / E`. Финальный RICE = среднее из трёх.

## Голоса

| Агент              | На что смотрит                                            |
|--------------------|-----------------------------------------------------------|
| Architect          | Техническая сложность, зависимости, архитектурный риск   |
| System Analyst     | Пользовательская ценность, бизнес-дифференциация         |
| Devil's Advocate   | Скептический взгляд: риски переоценки, скрытые усложнения |

Для всех трёх — `model: opus` (по правилам проекта).

## Workflow через Agents Team

В одном сообщении из главного Claude (после открытия `tmux-team.sh`):

```
1. TeamCreate(name="rice-F-042")
2. TaskCreate × 3 (по одной задаче на агента)
3. Task × 3 одновременно с team_name="rice-F-042":
   - subagent_type="architect"          model="opus"
   - subagent_type="system-analyst"     model="opus"
   - subagent_type="devils-advocate"    model="opus"
   prompt — одинаковый: «Оцените F-042 по RICE …», + полный контекст Issue.
4. По мере готовности: TaskList → собрать результаты.
5. Заполнить таблицу в описании Issue (см. шаблон ниже).
6. TeamDelete.
```

## Шаблон таблицы в Issue

```markdown
## RICE-оценка

| Голос              | Reach | Impact | Confidence | Effort (PM) | RICE  |
|--------------------|------:|-------:|-----------:|------------:|------:|
| Architect          |     4 |    1.0 |        0.8 |         1.5 |  2.13 |
| System Analyst     |     4 |    2.0 |        0.7 |         2.0 |  2.80 |
| Devil's Advocate   |     3 |    1.0 |        0.5 |         3.0 |  0.50 |
| **Среднее**        |   3.7 |   1.33 |       0.67 |        2.17 |  **1.81** |

### Обоснование
- **Architect:** упрощает миграцию данных, риски в FK constraints. ~6 недель.
- **System Analyst:** влияет на 60% активных пользователей (CRM-роль).
- **Devil's Advocate:** есть риск scope creep — корзина и checkout могут потребовать рефакторинга.
```

## Промпт для агента (можно вставлять сразу из CLAUDE.md)

```
Оцените Issue {{ID}} по RICE.

Контекст Issue:
{{полный текст из ./scripts/issue-list.sh ...}}

Выведите ответ в формате:
- Reach: <1-5>
- Impact: <0.25 | 0.5 | 1 | 2 | 3>
- Confidence: <0-1>
- Effort: <person-months>
- RICE: <вычислите>
- Обоснование: 2-3 предложения. Не отклоняйтесь — только то, что соответствует вашей роли.
```

## Когда RICE необязателен

- Багфикс с очевидным приоритетом (P0/P1) — указать только severity.
- Тривиальный TD-тикет (< 1 час) — пропустить, отметить `effort: trivial`.
- Безопасность с CVE — пропустить, исполнить как hotfix.

В остальных случаях RICE — обязательная часть описания Issue.
