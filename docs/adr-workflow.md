# ADR Workflow (Wiki как хранилище)

**ADR (Architecture Decision Record)** — формальная запись принятого
архитектурного или системно-аналитического решения. Хранятся в **Wiki проекта**
(GitLab Wiki / GitHub Wiki), не в коде.

## Когда ADR обязателен

ADR создаётся, если затрагивается хотя бы один из пунктов:

1. **Архитектура** — новый компонент, изменение слоистости, выбор паттерна
   (CQRS, event-sourcing, saga, hexagonal и т. п.).
2. **Технологии** — выбор/замена БД, кеша, брокера, ORM, фреймворка, runtime.
3. **API** — новый внешний API, breaking change в существующем,
   формат идемпотентности, RBAC-модель.
4. **Схема данных** — новая основная сущность, нетривиальные связи (M:N
   с историей, soft-delete, шардирование), миграции, влияющие на downstream.
5. **Интеграции** — новый внешний сервис, протокол обмена, retry-стратегия.
6. **Безопасность** — auth-схема, секрет-менеджмент, threat model.
7. **Performance/Scaling** — стратегии кэширования, sharding, read-replica.

## Кто отвечает

- **Architect** — обязан создать ADR при любом архитектурном решении.
- **System Analyst** — обязан создать ADR при формализации спецификации,
  если спецификация фиксирует non-trivial выбор (модель данных, контракт API,
  бизнес-правило с альтернативами).

ADR создаётся **до** или **синхронно с** реализацией, не после.

## Структура ADR

```markdown
# ADR-NNN: Краткое название решения

## Status
Proposed | Accepted | Deprecated | Superseded by ADR-MMM

Date: YYYY-MM-DD
Authors: <agent-role или ник>
Reviewers: <DA / другие участники>

## Context
Описание проблемы, ограничений, мотивации. Что заставляет принимать решение
именно сейчас? Какие силы действуют (бизнес, техдолг, compliance)?

## Decision
Описание принятого решения чёткой формулировкой. Что мы выбрали и как
это будет реализовано на верхнем уровне.

## Consequences

### Положительные
- ...

### Отрицательные
- ...

### Риски и митигация
- Риск → как митигируем

## Alternatives Considered
Какие варианты рассматривались. По каждому: 1-2 строки + причина отказа.

| Вариант | Плюсы | Минусы | Почему отклонён |
|---------|-------|--------|------------------|

## Links
- Связанные ADR: ADR-XXX (зависит), ADR-YYY (заменено этим)
- Issues: F-NNN, B-MMM
- MR/PR: !NNN
- Внешние ссылки на статьи/блог-посты
```

## Нумерация

`ADR-001`, `ADR-002`, ... — сквозная, без пропусков. Получить следующий номер:

```bash
./scripts/adr-create.sh --next-number
# -> 042
```

## Index-страница в Wiki

Wiki всегда содержит страницу `ADR-Index` (или `Home` со ссылкой),
которая перечисляет все ADR в порядке номеров с их статусом:

```markdown
# ADR Index

| #     | Название                        | Status     | Date       |
|-------|----------------------------------|------------|------------|
| 001   | Layered architecture             | Accepted   | 2026-01-15 |
| 002   | PostgreSQL вместо MongoDB        | Accepted   | 2026-01-20 |
| 003   | JWT с rotating refresh tokens    | Superseded | 2026-02-01 |
| 004   | Refresh tokens через cookies     | Accepted   | 2026-02-10 |
```

Index-страница обновляется одновременно с созданием ADR (или скриптом).

## Создание ADR

```bash
# Шаблон в stdout, отредактировать и сохранить
./scripts/adr-create.sh "Выбор брокера: NATS vs Kafka" > /tmp/adr.md
$EDITOR /tmp/adr.md

# Опубликовать в Wiki проекта
./scripts/adr-create.sh --publish "Выбор брокера: NATS vs Kafka" /tmp/adr.md
```

Скрипт работает и с GitLab Wiki (через `glab api`), и с GitHub Wiki
(через клон wiki git-репозитория).

## Цикл изменения существующего ADR

ADR — **immutable** после Accepted. Если решение меняется:
1. Старый ADR переводится в `Status: Superseded by ADR-MMM`.
2. Создаётся новый ADR-MMM с `Status: Accepted`, в `Context` ссылка на ADR-NNN.
3. Index-страница обновляется.

Не редактируем тело принятого ADR — это история, к которой обращаются
через 6 месяцев, чтобы понять, почему так было.

## Кросс-ссылки в коде/Issues

- В коде, если решение неочевидно: `// ADR-042: NATS выбран из-за at-most-once семантики`.
- В Issues, относящихся к решению: ссылка на Wiki-страницу.
- В MR description: `Implements ADR-042`.
- В CHANGELOG: ссылка на ADR при breaking change.

## Чек-лист перед закрытием Issue

Если в Issue был принят выбор технологии/паттерна — закрытие Issue **блокируется**
до публикации ADR.

- [ ] Создана страница `ADR-NNN: <title>` в Wiki.
- [ ] `Status: Accepted` (не `Proposed`, если уже мерджим).
- [ ] Index-страница обновлена.
- [ ] Ссылка на ADR в комментарии Issue.
- [ ] Ссылка на ADR в описании MR.
