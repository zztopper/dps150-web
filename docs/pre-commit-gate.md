# Pre-Commit Gate

**Жёсткое правило:** до `git commit` обязана пройти **полная локальная
проверка** того, что:

1. Код собирается.
2. Все тесты пирамиды зелёные локально.
3. LikeC4-модель валидна.
4. Реализация **полно** покрывает ТЗ (Issue + AC).
5. Devil's Advocate провёл ревью, все находки устранены/зафиксированы.

Если хотя бы один пункт красный — коммит запрещён, исправляем и проверяем
заново. Никаких «закоммичу, в CI поправлю» — это размывает ответственность
и рушит trunk.

## Полный чек-лист

### A. Сборка
- [ ] **Backend** собирается без ошибок: `make build` (или эквивалент: `go build`,
      `cargo build --release`, `mvn package`, `npm run build`).
- [ ] **Frontend** собирается без ошибок: `make build-frontend` (или `npm run build`,
      `vite build`, `pnpm build`).
- [ ] **Docker-образы** собираются: `make build-docker`. Это ловит проблемы,
      специфичные для Dockerfile (alpine-несовместимости, права, отсутствующие deps).

### B. Линтеры и формат
- [ ] `make lint` — без warnings (treat-as-error, не игнорировать).
- [ ] Форматтер прошёл (`gofmt`/`prettier`/`black`/etc.) — никаких diff'ов.
- [ ] Type-check (`tsc -b` / `mypy` / etc.) — зелёный.

### C. Пирамида тестов
Все три уровня зелёные локально (см. `docs/testing-pyramid.md`):
- [ ] **Unit:** `make test` или `make test-cover` — зелёный.
- [ ] **Coverage ~60 %** на core/handlers (ориентир, без догмы).
- [ ] **Integration:** `make test-int` — зелёный (testcontainers / реальная БД).
- [ ] **E2E:** `make test-e2e` (Playwright/Cypress) — зелёный для smoke + happy
      ключевой фичи. Если фича не user-видимая — пропустить и явно отметить.
- [ ] Тесты не flaky: повторный прогон зелёный.

### D. LikeC4
- [ ] Если затронута архитектура / новый флоу — `.c4` файлы обновлены
      (см. `docs/likec4-workflow.md`).
- [ ] **`npx likec4 validate docs/architecture/likec4`** — без ошибок.
      Это **обязательная** проверка в pre-commit gate.

### E. ADR
- [ ] Если принято архитектурное решение — ADR опубликован в Wiki
      (см. `docs/adr-workflow.md`).
- [ ] Ссылка на ADR добавлена в описание Issue/MR.

### F. Соответствие ТЗ
- [ ] Все Acceptance Criteria из Issue реализованы (либо явно отложены —
      с TD-Issue и обоснованием).
- [ ] Out of scope из Issue не нарушен (нет «бонус-фич», не согласованных
      с оркестратором).
- [ ] Side-effects, миграции, breaking changes — отражены в:
      - CHANGELOG.md (`[Unreleased]`);
      - Swagger/OpenAPI (если backend);
      - Wiki (ADR, если архитектурное);
      - LikeC4 (если архитектурное / флоу).

### G. Devil's Advocate
- [ ] DA-Task запущен в Agents Team.
- [ ] Отчёт получен.
- [ ] CRITICAL и HIGH — устранены полностью **в текущей сессии**.
- [ ] **Каждая** не-устранённая находка (MEDIUM/LOW/WISHES, pre-existing,
      out-of-scope, любой другой случай) — оформлена как **новый Issue**
      в трекере. Без исключений.
- [ ] Все Issue-ссылки указаны в DA-отчёте (секция «Tracked»).
- [ ] Verdict DA: **APPROVED**.

> Принцип: «находка без следа» запрещена. Если что-то не делаем сейчас —
> для этого есть Issue, чтобы аудит-след сохранился. Никаких устных «отложили».

### H. Issue & ветка
- [ ] Issue в статусе In Progress, owner — текущий разработчик.
- [ ] Имя ветки: `feature/F-NNN-...` / `bugfix/B-NNN-...` / `tech-debt/TD-NNN-...`.
- [ ] Возраст ветки ≤ 3 дня (если больше — декомпозиция Issue провалена).

## Скрипт-обёртка

`./scripts/precommit-check.sh` запускает A–D автоматически.
E–H — отвечает разработчик/оркестрирующий агент (это содержательные проверки, не bash).

```bash
./scripts/precommit-check.sh           # запустить весь автоматический набор
./scripts/precommit-check.sh --quick   # пропустить E2E и docker build (для частых правок)
./scripts/precommit-check.sh --full    # дополнительно security scans, dependency audit
```

При красном результате коммит блокируется (скрипт возвращает non-zero).

## Рекомендуемый git-hook (опционально)

`.git/hooks/pre-commit`:
```bash
#!/usr/bin/env bash
exec ./scripts/precommit-check.sh --quick
```

`chmod +x .git/hooks/pre-commit`.

Hook не панацея: разработчик всё равно отвечает за E–H. Но он
отлавливает «забыл запустить тесты» как минимум.

## Анти-паттерны

- ❌ «CI поймает» — в TBD это разрушает master. CI — это safety net,
  а не первая линия защиты.
- ❌ Закоммитить «WIP», потом fixup — повышает риск, что fixup забудут,
  и красное войдёт в master.
- ❌ Пропустить E2E «потому что долго» — добавьте `--quick` режим
  для частых коммитов в ветке, но **перед push** запускайте полный набор.
- ❌ Скип `likec4 validate` «потому что он у меня не сломан» — модель
  ломается часто из-за рефакторинга id/имён.
- ❌ DA только в самом конце — это поздно. DA должен ревьюить
  каждый значимый промежуточный коммит, минимум — финальный.

## Что разрешено пропустить (с обоснованием)

| Шаг             | Пропуск разрешён, если                                              |
|-----------------|----------------------------------------------------------------------|
| E2E             | Изменение чисто backend без user-видимого поведения                  |
| Integration     | Изменение чисто во фронтенде/UI без backend-связи                    |
| Docker build    | Изменение исключительно в `docs/`/комментариях                       |
| LikeC4          | Изменение не затрагивает архитектуру/флоу (косметика, тесты)         |
| ADR             | Не принято новое архитектурное решение                              |

Все пропуски — с явной отметкой в commit-message: `[skip e2e — backend-only]`.
DA в ревью подтверждает корректность пропуска или разворачивает.
