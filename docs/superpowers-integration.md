# Superpowers + мультиагентная оркестрация

Claude Code поставляется с набором персональных skills (часто называемых
«superpowers»), которые живут в `~/.claude/skills/`. Они полезны, но
часть из них спавнит субагенты через Task tool.
В этом проекте действует
**жёсткое правило: все subagent-вызовы — только через Agents Team**
(см. `docs/agents-team-workflow.md`).

Этот документ объясняет, как использовать superpowers, не нарушая правило.

## Какие superpowers применимы (и как)

| Superpower                          | Где полезна                                                     | Совместимо с правилом?                                  |
|-------------------------------------|------------------------------------------------------------------|----------------------------------------------------------|
| `brainstorming`                     | Начало планирования фичи, поиск альтернатив                      | ✅ напрямую (без Task)                                    |
| `writing-plans`                     | Перевод Issue в imp-план перед TeamCreate                       | ✅ напрямую                                               |
| `test-driven-development`           | Backend/Frontend agents следуют TDD-циклу                       | ✅ напрямую внутри субагента                               |
| `systematic-debugging`              | DA / SDE при отладке                                             | ✅ напрямую                                               |
| `executing-plans`                   | После согласования плана — запуск через TeamCreate              | ✅ при условии Wrap (см. ниже)                            |
| `verification-before-completion`    | Финальная самопроверка перед коммитом                           | ✅ напрямую (DA остаётся обязательным шагом сверху)       |
| `dispatching-parallel-agents`       | Параллельное распределение работы                               | ⚠️  **только обёрнутая** — см. ниже                        |

## Правило-обёртка

Любой superpower-skill, который **может вызвать `Task` или
`run_in_background`**, обязан в этом проекте быть запущен только после того,
как уже создана активная Agents Team:

1. Создаёшь команду: `TeamCreate({ name: "<short-id>" })`.
2. Используешь skill, передавая ему контекст команды:
   ```
   Skill: dispatching-parallel-agents
   Args: --team_name=<short-id>  (или эквивалент)
   ```
3. Если skill создаёт Task без `team_name` — **хук
   `enforce-agents-team.sh` его заблокирует** (см. `.claude/hooks/`).
   Это намеренно: гарантия защитного барьера.

## Гейтинг hooks

В `.claude/settings.json` настроены `PreToolUse`-хуки:

- **Task** → `enforce-agents-team.sh`. Блокирует:
  - `Task` без `team_name` (exit 2);
  - `Task` с `run_in_background=true` (exit 2).
- **Skill** → `check-superpowers-task.sh`. Soft-warning (без блока) для
  skills, имена которых указывают на спавн агентов.

Хуки применяются **ко всем** запускам Claude Code в этом проекте, включая
запуски тимлида и teammates.

## Запреты

- ❌ **Background-агенты.** `Task({ run_in_background: true })` запрещены.
  Причина: обходят tmux, обходят DA-гейт, нарушают предсказуемость.
- ❌ **Solo Task без TeamCreate.** Даже одиночный subagent-вызов оборачивается
  в TeamCreate (один teammate — это валидная «команда из одного»).
  Причина: единая семантика наблюдаемости и истории.
- ❌ **Skill, который сам не подчиняется правилу.** Если superpower
  непосредственно нарушает гейт (попытается один из запрещённых
  вариантов `Task`) — он будет автоматически заблокирован хуком.
  Не пытайтесь обойти хук.

## Memory precedence

Память в проекте живёт в трёх местах. При конфликте действует приоритет:

1. **Реальный код / git history** — единственная истина в последней инстанции.
   Если память противоречит коду, доверяем коду и обновляем память.
2. **`.claude/memory/MEMORY.md`** — auto-memory Claude Code, индекс плюс
   файлы. Главное место для пользовательских предпочтений и проектных фактов,
   которые не выводятся из кода.
3. **`.serena/memories/`** — заметки Serena о коде / архитектуре. Полезны,
   но более узкие.
4. **`.claude/memory/mcp-memory.jsonl`** (memory MCP) — структурированный
   knowledge graph, доступный через MCP-tools. Удобен для агентов, но не
   читается человеком напрямую.

Правила:
- Перед действием на основе памяти — **верифицировать** в коде или внешней
  системе (`git log`, `glab`/`gh`, реальный файл).
- При обновлении: пишите в самый специфичный слой (если про код — Serena;
  если про процессы/предпочтения — MEMORY.md; если про граф связей —
  MCP memory).
- Дубликаты — удалять. Один факт — одно место.

## Skills и slash-команды (что использовать руками)

| Команда                  | Когда использует оркестратор                                    |
|---------------------------|------------------------------------------------------------------|
| `/init`                   | Начальная инициализация CLAUDE.md из codebase (опционально)      |
| `/review`                 | Дополнительный ревью **в дополнение** к DA-агенту, не вместо     |
| `/security-review`        | Перед merge security-чувствительной ветки                       |
| `/simplify`               | Перед DA — самопроверка SDE-агента                               |
| `/loop`                   | Только для long-running внешних проверок (CI poll), не агентов   |

## FAQ

**Q: Я делаю одиночный исследовательский Task. Тоже надо TeamCreate?**
A: Да. Создайте команду на одного teammate; добавьте Task с этим team_name.
   После завершения — TeamDelete. Накладные расходы минимальны, выгода —
   единая семантика и наблюдаемость.

**Q: Я хочу запустить агент в фоне, чтобы вернуться через час.**
A: Запрещено хук-ом. Альтернатива: используйте `/loop` для опроса
   состояния long-running CI-задачи (бэш-команда, не Task), или
   разбейте работу на синхронные шаги.

**Q: Хук блокирует мой workflow, а у меня горит.**
A: Не отключайте хук. Скорее всего, вы пытаетесь обойти процесс. Лучше:
   быстро сделайте `TeamCreate({name:"hotfix"})`, потом Task с этим именем.
   Это занимает 2 секунды и даёт правильную траекторию.
